# Architecture

CJA is organized as three subsystems — Cron, Jobs, and Axum (HTTP server) — plus supporting modules for sessions, tracing, and database management. Everything coordinates through a shared `AppState` trait and `CancellationToken` for graceful shutdown.

## Crate Structure

```
Cargo.toml          — Workspace root, clippy/lint config
crates/
  cja/              — Core framework library
  cja.app/          — Example application (canonical usage reference)
  cli/              — `cja` CLI tool (project scaffolding)
migrations/         — Copy of crates/cja/migrations/ for sqlx CLI
```

### Module Tree (`crates/cja/src/`)

```
lib.rs             — Re-exports (sqlx, uuid, color_eyre, maud, chrono, chrono_tz)
app_state.rs       — AppState trait
setup.rs           — Sentry + tracing initialization
db.rs              — run_migrations() via sqlx::migrate!("./migrations")
tasks.rs           — NamedTask, wait_for_first_error
jobs/
  mod.rs           — Job trait, enqueue logic, constants (DEFAULT_MAX_RETRIES, DEFAULT_LOCK_TIMEOUT)
  worker.rs        — job_worker() polling loop, retry/dead-letter logic, FOR UPDATE SKIP LOCKED
  registry.rs      — JobRegistry trait, impl_job_registry! macro
cron/
  mod.rs           — Re-exports
  registry.rs      — CronRegistry, Schedule enum (Interval | Cron), CronJob
  worker.rs        — Cron Worker tick loop with configurable poll interval and timezone
server/
  mod.rs           — run_server(), middleware stack (tracing + cookies)
  cookies/
    cookie_key.rs  — CookieKey (from COOKIE_KEY env or auto-generated)
    cookie_jar.rs  — CookieJar Axum extractor
  session.rs       — CJASession, AppSession trait, Session<A> extractor
  page.rs          — Page wrapper for HTML responses
  page/factory.rs  — Factory Axum extractor
  trace.rs         — Tracer (MakeSpan + OnResponse for tower-http)
testing/
  mod.rs           — Feature-gated test utilities
  mock_oauth/      — Mock OAuth server for integration tests
```

## Key Traits

### `AppState` (`app_state.rs`)

Central trait all applications must implement. Provides access to the database pool, cookie key, and version string. Cloned per-request — wrap expensive resources in `Arc`.

```rust
pub trait AppState: Clone + Send + Sync + 'static {
    fn version(&self) -> &str;
    fn db(&self) -> &sqlx::PgPool;
    fn cookie_key(&self) -> &CookieKey;
}
```

### `Job<AS>` (`jobs/mod.rs`)

Defines a background job type. Key members:
- `NAME: &'static str` — unique identifier used for routing
- `run(&self, app_state) -> Result<()>` — job logic
- `run_with_cancellation(&self, app_state, token) -> Result<()>` — override for graceful shutdown support
- `enqueue(self, app_state, context, priority) -> Result<()>` — inserts job into the `jobs` table

### `JobRegistry<AS>` (`jobs/registry.rs`)

Dispatches jobs by name to their handler. Generated by the `impl_job_registry!` macro:

```rust
cja::impl_job_registry!(AppState, NoopJob, CancellableDemoJob);
// Generates a `Jobs` struct implementing JobRegistry
```

### `AppSession` (`server/session.rs`)

Trait for custom session types backed by the `sessions` table:
- `from_db(pool, session_id) -> Result<Self>`
- `create(pool) -> Result<Self>`
- `from_inner(CJASession) -> Self`
- `inner(&self) -> &CJASession`

The `Session<A>` extractor automatically loads or creates sessions from the `session_id` cookie.

## Data Flow

### HTTP Request Flow

```
Incoming request
  → TcpListener (or systemfd handoff)
    → tower-http TraceLayer (MakeSpan + OnResponse)
      → CookieManagerLayer
        → Axum Router (your routes)
          → Handler (with extractors: Session, CookieJar, State, etc.)
            → Response
```

### Job Flow

```
enqueue() → INSERT INTO jobs (with priority, payload, context)
  → Worker poll loop (every sleep_duration)
    → SELECT ... FOR UPDATE SKIP LOCKED (locks one row)
      → Deserialize payload → JobRegistry::run_job() → Job::run()
        → On success: DELETE FROM jobs
        → On failure (error_count < max_retries):
            UPDATE jobs SET error_count += 1,
              run_at = NOW() + 2^(error_count+1) seconds
        → On failure (error_count >= max_retries):
            INSERT INTO dead_letter_jobs + DELETE FROM jobs
```

The `FOR UPDATE SKIP LOCKED` clause is what makes multiple concurrent workers safe — each worker atomically claims a different row. Priority ordering is `ORDER BY priority DESC, created_at ASC` (higher priority value = runs first).

### Cron Flow

```
CronRegistry (job name → Schedule + enqueue function)
  → Worker tick loop (every sleep_duration, default 60s)
    → For each registered job:
      → Check Schedule::should_run() against last_run in crons table
        → If due: enqueue the job, update last_run
```

Cron jobs are regular `Job` implementations — the cron system just handles scheduling the enqueue.

## Database Schema

### `jobs`

| Column | Type | Description |
|--------|------|-------------|
| `job_id` | `UUID PRIMARY KEY` | Unique job identifier |
| `name` | `TEXT NOT NULL` | Job type name (matches `Job::NAME`) |
| `payload` | `JSONB NOT NULL` | Serialized job data |
| `priority` | `INT NOT NULL` | Higher value = runs first (default 0) |
| `run_at` | `TIMESTAMPTZ NOT NULL` | When the job becomes eligible to run |
| `created_at` | `TIMESTAMPTZ NOT NULL` | When the job was enqueued |
| `locked_at` | `TIMESTAMPTZ` | When a worker claimed this job |
| `locked_by` | `TEXT` | Worker ID holding the lock |
| `context` | `TEXT NOT NULL` | Human-readable enqueue context |
| `error_count` | `INTEGER NOT NULL DEFAULT 0` | Number of failed attempts |
| `last_error_message` | `TEXT` | Most recent error |
| `last_failed_at` | `TIMESTAMPTZ` | Most recent failure timestamp |

### `dead_letter_jobs`

| Column | Type | Description |
|--------|------|-------------|
| `id` | `UUID PRIMARY KEY DEFAULT gen_random_uuid()` | Row ID |
| `original_job_id` | `UUID NOT NULL` | Original job's ID |
| `name` | `TEXT NOT NULL` | Job type name |
| `payload` | `JSONB NOT NULL` | Serialized job data |
| `context` | `TEXT NOT NULL` | Enqueue context |
| `priority` | `INT NOT NULL` | Original priority |
| `error_count` | `INTEGER NOT NULL` | Total failure count |
| `last_error_message` | `TEXT` | Final error message |
| `created_at` | `TIMESTAMPTZ NOT NULL` | Original creation time |
| `failed_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | When moved to dead letter |

### `crons`

| Column | Type | Description |
|--------|------|-------------|
| `cron_id` | `UUID PRIMARY KEY` | Cron entry ID |
| `name` | `TEXT NOT NULL` | Unique cron job name (unique index) |
| `last_run_at` | `TIMESTAMPTZ NOT NULL` | Last execution time |
| `created_at` | `TIMESTAMPTZ NOT NULL` | Entry creation time |
| `updated_at` | `TIMESTAMPTZ NOT NULL` | Last update time |

### `sessions`

| Column | Type | Description |
|--------|------|-------------|
| `session_id` | `UUID PRIMARY KEY DEFAULT gen_random_uuid()` | Session ID |
| `created_at` | `TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP` | Creation time |
| `updated_at` | `TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP` | Auto-updated via trigger |

The `sessions` table has an `update_updated_at_column()` trigger that automatically sets `updated_at` on updates.

## Graceful Shutdown

Shutdown flows through `tokio_util::sync::CancellationToken`:

1. Signal handler (SIGINT/SIGTERM) calls `token.cancel()`
2. All subsystems (`job_worker`, cron `Worker::run`, etc.) check `token.cancelled()` in their `tokio::select!` loops
3. Workers finish their current work item, then exit
4. Job worker runs `cleanup_worker_locks()` to release any held locks
5. `wait_for_first_error()` completes when the first task exits
6. Application shuts down

## Migration Locations

Three copies of migrations exist in the repository:

| Location | Purpose | Used by |
|----------|---------|---------|
| `crates/cja/migrations/` | **Source of truth** | `cja::db::run_migrations()` via `sqlx::migrate!("./migrations")` |
| `migrations/` (root) | Copy for `sqlx` CLI | `sqlx migrate run` from workspace root |
| `crates/cja.app/migrations/` | Example app's copy | `sqlx::migrate!()` in `cja.app/src/main.rs` |

When adding a new framework migration, it must be added to all three locations. The `sqlx::migrate!("./migrations")` macro in `db.rs` resolves the path relative to the crate's `Cargo.toml`, i.e., `crates/cja/migrations/`. The `sqlx::migrate!()` call (no path argument) in `cja.app` similarly resolves to `crates/cja.app/migrations/`.
